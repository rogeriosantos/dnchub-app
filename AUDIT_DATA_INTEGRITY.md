## Findings

| # | Severity | File | Line(s) | Description | Recommendation |
|---|----------|------|---------|-------------|----------------|
| 1 | CRITICAL | `transformers.ts` | 601–622 | **`conditionAtCheckout` and `conditionAtReturn` are cast unsafely from string.** `ToolAssignmentResponse` types these fields as `string \| null`, but the transformer casts them directly as `ToolAssignment["conditionAtCheckout"]` (a union of `ToolCondition \| null`). If the API returns an unexpected string value, the cast silently produces an invalid enum value. Downstream in `inventory/[id]/page.tsx` (line 516), that value is used as an index into `toolConditionConfig` without guarding, which throws a runtime error: `Cannot read properties of undefined`. | Validate the string against the allowed `ToolCondition` union before casting; return `null` on unknown values. |
| 2 | CRITICAL | `lib/api/client.ts` | 131–150 | **Race condition in token refresh for concurrent 401 responses.** `isRefreshing` is a module-level boolean, but `refreshPromise` is set and immediately awaited in the same microtask. If two requests fail with 401 simultaneously, the second enters the `!isRefreshing` guard as `false`, but `refreshPromise` may still be `null` at the point the first request sets `isRefreshing = true` and before it assigns `refreshPromise`. The second caller then awaits `null`, which resolves to `null` (falsy), causing it to clear tokens and redirect to `/login` even though a valid refresh was in flight. | Move the `refreshPromise` assignment before setting `isRefreshing = true`, and null-check `refreshPromise` before awaiting it. |
| 3 | CRITICAL | `tools/assignments/page.tsx` | 392–450 | **Assignment form submits without validating that a `vehicle` or `employee` ID is resolved.** `handleAssignSubmit` checks `assignForm.assignedToEmployeeId` and `assignForm.assignedToVehicleId` presence for those assignment types, but for `department` and `section` types there is no length/trim validation. An empty string `""` is sent as the `department` or `section` payload, which may be stored as an empty value in the backend, silently creating corrupt assignment records. | Add `.trim().length > 0` validation for `department` and `section` fields before calling the API. |
| 4 | CRITICAL | `tools/inventory/[id]/page.tsx` | 86–91 | **Tool detail page fetches only active assignments to build the full assignment history tab.** `toolAssignmentsService.getActive()` is called (line 87), but the "Assignment History" tab (line 473) renders **all** `assignments`, implying both active and historical records. Historical (returned) assignments are silently absent from the tab. The user sees an incomplete history, and the count shown in the tab badge is wrong. | Call `toolAssignmentsService.getAll({ tool_id: toolId })` for the history tab, and separately derive the active assignment from the returned list. |
| 5 | CRITICAL | `transformers.ts` | 680–695 | **`toSnakeCase` does not recurse into nested objects or arrays.** All `create` and `update` service methods pass form data through `toSnakeCase` before sending it to the API. If any field is an object or array (e.g., `images: string[]` on `ToolCase` / `Tool`), the nested keys remain in camelCase. The backend will reject or silently ignore them, causing silent data loss on `images` arrays. | Add recursive handling for nested objects and arrays in `toSnakeCase`. |
| 6 | HIGH | `types/index.ts` + `lib/api/types.ts` | 559–579 / 895–919 | **`Tool.organizationId` is missing from the frontend domain type.** `ToolResponse` (API type) has `organization_id: string` (required), but the `Tool` interface in `types/index.ts` does not include `organizationId`. This field is silently dropped in `transformTool`. If any component or service later needs to filter by org, it will fail silently. `ToolCase` has the same problem (no `organizationId` in domain type). | Add `organizationId: string` to both `Tool` and `ToolCase` domain interfaces, and map it in the respective transformers. |
| 7 | HIGH | `transformers.ts` | 569–594 | **`transformTool` does not include `organizationId`** (matching issue #6). The field is available in `ToolResponse.organization_id` but is not mapped in the transformer body. | Map `organizationId: api.organization_id` in `transformTool` and `transformToolCase`. |
| 8 | HIGH | `tools/consumables/page.tsx` | 117–127 | **`handleAdjustConfirm` silently swallows errors.** The `catch` block is empty (`catch { /* handled by mutation */ }`). If `adjustMutation.mutate` throws (e.g., 422 validation or 500), the dialog stays open in a non-loading state with no error message shown to the user. The same pattern exists in `handleDeleteConfirm` (line 109–116). | Surface the error via state (`setError`) or a toast so the user knows the action failed. |
| 9 | HIGH | `tools/inventory/page.tsx` | 107–127 | **Status override logic has a logical gap.** `effectiveTools` overrides a tool's status to `'assigned'` only if `tool.status === 'available'`. A tool with status `'in_repair'` that also has an active assignment is not overridden, so the tool appears as `in_repair` even though it is out on assignment. The stats card for "Assigned" will under-count. | Remove the `tool.status === 'available'` guard, or apply the override for any non-terminal status. |
| 10 | HIGH | `tools/assignments/page.tsx` | 354–373 | **`handleReturnConfirm` ignores the per-tool condition checklist for case tools.** The return dialog collects conditions for each tool inside a case (`returnChecklist`), but `handleReturnConfirm` only sends `condition_at_return` from the **main** item key. The individual tool conditions gathered in the checklist are discarded. | Either call a per-tool return API endpoint for each checked item, or document clearly that only the case-level condition is persisted. |
| 11 | HIGH | `tools/categories/page.tsx` | 121–139 | **Category save does not prevent self-reference.** When editing, the parent category dropdown correctly filters `c.id !== editingCategory?.id`, but the filter only removes a single level. A user can create a cycle: A → B → A. The `filteredCategories` selector uses `getParentName` which would infinitely recurse if the UI ever renders a tree, and the API may or may not validate this. | Collect and exclude all descendants of the editing category from the parent dropdown. |
| 12 | HIGH | `tools/assignments/page.tsx` | 128–130 | **`assignMutation` cast bypasses type safety.** The mutation parameter is typed as `Record<string, unknown>` and cast to `Partial<ToolAssignment>` with `as`. This means `assigned_by_id` (a required field in `ToolAssignmentResponse`) is never set in the outgoing payload. The backend requires it and will return a 422; the UI shows no specific error because errors are only caught at the console level. | Include `assignedById` from the authenticated user context in the form payload before submission. |
| 13 | HIGH | `tools/inventory/[id]/page.tsx` | 110–112 | **Delete failure is silently swallowed.** `handleDelete` catches the error and only resets `isDeleting` to `false`. The user sees no toast or error message; the UI appears to silently do nothing. | Display an error alert or toast on delete failure. |
| 14 | MEDIUM | `lib/api/client.ts` | 154–171 | **Non-ok responses with non-JSON bodies throw `parse_error` which hides the real HTTP status meaning.** A 404 from the tool detail endpoint will surface as `"parse_error: Request failed with status 404"` if the body is HTML (e.g., a proxy returning an HTML error page). The tools detail page then shows `'Tool not found'` incorrectly for all network errors. | Check `Content-Type` response header before parsing; if not JSON, create the error directly from the HTTP status/statusText. |
| 15 | MEDIUM | `lib/api/tools.ts` | 92–104 | **`toolCategoriesService.getAll` and `toolLocationsService.getAll` pass `search` param directly from `ListParams` without sanitization.** An empty string `""` is filtered by the `buildUrl` function (`value !== ""`), which is correct, but the `skip` and `limit` params can be `0` and would be included. A `limit=0` sent to the backend may return zero results, breaking list pages silently. | Explicitly exclude `0` or validate that `limit` is a positive integer before adding it to query params. |
| 16 | MEDIUM | `tools/consumables/[id]/page.tsx` | 48–54 | **`statusConfig` uses hardcoded English label strings instead of i18n keys.** Unlike the consumables list page (`consumables/page.tsx`) which uses `labelKey` pointing to i18n translations, the detail page uses raw English strings (`'In Stock'`, `'Low Stock'`, etc.). This breaks localization consistency — if the app is used in another language, the detail page shows English while the list page is translated. | Unify to use translation keys with `useTranslation`, consistent with the list page. |
| 17 | MEDIUM | `tools/inventory/page.tsx` | 82–84 | **Tools list is fetched with a hardcoded `limit: 2000`.** This is a client-side pagination workaround that could cause significant memory and rendering performance issues with large inventories. The same pattern appears in `tools/categories/page.tsx` (line 63). If the actual tool count exceeds 2000, results are silently truncated. | Implement server-side pagination or at minimum display a warning when the returned count equals the limit. |
| 18 | MEDIUM | `tools/inventory/new/page.tsx` | 67–98 | **Form validation is entirely delegated to HTML `required` attributes.** There is no client-side validation that `erpCode` is unique, that `calibrationIntervalDays` is a positive integer, or that `purchasePrice` is non-negative before submission. Errors only surface after the API call fails, with a generic `err.message` shown. | Add pre-submission validation for at minimum: `erpCode` format, `calibrationIntervalDays > 0`, and `purchasePrice >= 0`. |
| 19 | MEDIUM | `tools/assignments/page.tsx` | 91–96 | **`assignments` is re-fetched when `viewFilter` changes, but there is no loading state shown during the refetch.** The previous data remains visible while the new fetch is in flight, potentially showing stale active-only data after switching to "all" view. | Show a loading overlay or skeleton during filter-triggered refetches, or clear state optimistically. |
| 20 | MEDIUM | `tools/inventory/[id]/page.tsx` | 104–108 | **Category and location are loaded with fire-and-forget `.then().catch(() => {})` calls.** If these secondary fetches fail, the UI silently shows "Not set" with no indication of a fetch error. If they succeed with stale data, there is no mechanism to refetch them. | Fetch these in the main `Promise.all` block so failures are part of the unified error state. |
| 21 | LOW | `lib/api/tools.ts` | 529–541 | **`consumablesService.getAll` does not pass the `unit` filter param** (`ConsumableListParams.unit`) to `queryParams`. It is declared in the interface but never added to the query object, silently ignoring any unit filter the caller specifies. | Add `unit: params?.unit` to the `queryParams` object in `consumablesService.getAll`. |
| 22 | LOW | `transformers.ts` | 548–562 | **`transformToolCase` drops `organization_id`** from `ToolCaseResponse`. Similar to finding #6/#7, if the org ID is ever needed on a `ToolCase` domain object it will be unavailable without a re-fetch. | Map `organizationId: api.organization_id` in `transformToolCase`. |
| 23 | LOW | `tools/cases/page.tsx` | 133–141 | **`filteredCases` returns `[]` when `effectiveCases` is empty but also when it has not yet loaded.** The condition `if (!effectiveCases.length)` treats "empty from API" the same as "not yet loaded", meaning an empty list after a successful fetch (0 cases) will display identically to a loading state. The early-return loading check above saves this for the initial load, but if data is cleared mid-session, it degrades incorrectly. | Check `effectiveCases` for `undefined`/null separately from a length of 0. |
| 24 | LOW | `data/mock-data.ts` | 888–936 | **`toolsNavigation` mock data is not imported by any tools page or dashboard component** in the audit scope. It is defined but unused in production paths. No tools mock data (tool objects, cases, assignments) was found in the file — no mock data leaks into production UI. The `toolsNavigation` export is risk-free but represents dead code. | Remove or relocate `toolsNavigation` to a navigation config module where it is actually consumed. |

## Notes on What Was NOT Found

- No mock tool/case/assignment/consumable objects were found in `mock-data.ts` — mock data does not leak into production API paths.
- The `apiClient` correctly handles 204 No Content, returning `undefined as T`.
- `transformToolCalibration` correctly preserves `cost: string | null` — no numeric coercion issue here since `ToolCalibration.cost` is typed `string | null` in both the domain and API types.
- All service delete methods (`toolsService.delete`, etc.) correctly return `Promise<void>` and do not attempt to parse the 204 response body.
